# Skill: interview-evaluator

## name
interview-evaluator

## description
Оценивает ответ кандидата на вопрос технического интервью по структурированной рубрике из 5 измерений. Возвращает числовую оценку, детальный разбор по каждому критерию, текстовый фидбек и флаг слабого ответа (`is_weak`), который используется для принятия решения о follow-up вопросе в LangGraph графе.

## triggers
Используй этот Skill когда:
- Нужно оценить ответ кандидата на технический или behavioral вопрос интервью
- Требуется консистентная многомерная оценка (не одна цифра, а структурированный разбор по критериям)
- Нужно определить — задавать follow-up или переходить к следующему вопросу (на основе `is_weak`)
- Нужен детальный фидбек для включения в финальный отчёт сессии

Ключевые слова-триггеры: "оцени ответ", "evaluate answer", "проверь ответ кандидата", "насколько хорошо ответил", "score this answer"

## when_not_to_use
Не используй этот Skill когда:
- Ответ — простое да/нет или числовой факт (достаточно обычного промпта)
- Вопрос ещё не задан (Skill работает только с парой вопрос + ответ)
- Нужна только транскрипция или перефраз ответа без оценки
- Оценивается не ответ кандидата, а качество самого вопроса

## input_format
```json
{
  "question": "строка — вопрос интервьюера",
  "answer": "строка — ответ кандидата",
  "role": "одно из: Backend Engineer | Frontend Engineer | ML Engineer | Data Analyst | Product Manager"
}
```

## output_format
Строго JSON, без лишнего текста вокруг:

```json
{
  "score": 7,
  "criteria": {
    "technical_correctness": 2,
    "clarity": 1,
    "depth": 2,
    "confidence": 1,
    "communication": 1
  },
  "reasoning": "краткое объяснение каждого критерия (1-2 предложения на каждый)",
  "feedback": "конструктивный фидбек для кандидата — что сделано хорошо и что улучшить",
  "is_weak": false
}
```

Правила вывода:
- `score` = сумма всех 5 criteria (диапазон 0–10)
- каждый критерий: `0` = отсутствует, `1` = частично, `2` = полностью раскрыто
- `is_weak` = `true` если `score < 6`
- `feedback` — на том же языке, что и ответ кандидата
- никакого текста за пределами JSON объекта

## example

**Вход:**
```json
{
  "question": "Объясни разницу между SQL и NoSQL базами данных",
  "answer": "SQL — реляционные БД с фиксированной схемой и ACID транзакциями, хорошо подходят для структурированных данных. NoSQL — гибкая схема, горизонтальное масштабирование, лучше для больших объёмов неструктурированных данных. Например, PostgreSQL vs MongoDB.",
  "role": "Backend Engineer"
}
```

**Выход:**
```json
{
  "score": 8,
  "criteria": {
    "technical_correctness": 2,
    "clarity": 2,
    "depth": 1,
    "confidence": 2,
    "communication": 1
  },
  "reasoning": "Technical correctness: верно определены ключевые различия — схема, транзакции, масштабирование. Clarity: ответ понятен и лаконичен. Depth: упомянуты ACID и масштабирование, но не раскрыты CAP теорема и конкретные use cases для выбора между типами. Confidence: кандидат отвечает уверенно, без оговорок. Communication: структура есть, но ответ мог быть чуть более последовательным (сначала критерий, потом пример).",
  "feedback": "Хороший ответ с правильными техническими фактами и примерами. Для усиления стоит добавить: в каких конкретных случаях выбирать SQL vs NoSQL (например, финансовые транзакции vs логи событий), и упомянуть CAP теорему как теоретическую основу trade-off между consistency и availability.",
  "is_weak": false
}
```

## notes

**Почему этот Skill лучше обычного промпта:**

Обычный промпт ("оцени ответ от 1 до 10") даёт одну цифру без объяснения. При прогоне одного и того же примера 3 раза подряд оценка может быть 6, 8, 7 — std ≈ 1.0. Кандидат не понимает что именно было слабым.

Этот Skill задаёт детерминированную рубрику из 5 конкретных критериев: агент не "думает абстрактно", а заполняет структурированные поля. Это решает две проблемы одновременно:
1. **Консистентность** — std при 3 прогонах снижается до ≈ 0.3 (подтверждено A/B тестом, см. EVALS.md)
2. **Actionability** — кандидат получает конкретный фидбек по каждому измерению, а не просто цифру

Поле `is_weak` напрямую используется как условие conditional edge в LangGraph графе: `is_weak=true AND follow_up_count < 2` → переход в ноду `follow_up_node`.
