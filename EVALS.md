# EVALS.md — InterviewPilot AI

## Обзор

Система оценки ответов кандидатов тестировалась на golden dataset из 30 примеров.
Проведён A/B эксперимент: два варианта промпта для `answer_evaluator`.

---

## Golden Dataset

**Размер:** 30 примеров  
**Роли:** Backend (8), Frontend (4), ML (6), Data Analyst (4), PM (4), DevOps (2), общие (2)  
**Типы ответов:**
- `strong` — развёрнутый экспертный ответ (ожидаемый score: 7–10)
- `weak` — поверхностный ответ без деталей (ожидаемый score: 1–5)
- `medium` — частично правильный ответ (ожидаемый score: 3–5)
- `dont_know` — кандидат не знает ответа (ожидаемый score: 0–2)
- `empty` — пустой ответ (ожидаемый score: 0–2)

Датасет покрывает edge cases: пустой ответ, ответ "не знаю", средний ответ с частичными знаниями.

## A/B Эксперимент

### Гипотеза
Промпт с детальной рубрикой (5 критериев × 2 балла) даёт более точные и стабильные оценки чем простой промпт "оцени от 0 до 10".

### Конфигурации

**Вариант A — простой промпт:**
```
Оцени ответ кандидата. Поставь оценку от 0 до 10. Дай краткий фидбек.
```

**Вариант B — промпт с рубрикой:**
```
Оцени по 5 критериям (каждый 0, 1 или 2):
- technical_correctness, clarity, depth, confidence, communication
```

### Результаты

| Метрика | Вариант A | Вариант B |
|---|---|---|
| Accuracy (в диапазоне) | **66.7%** (20/30) | 23.3% (7/30) |
| LLM-as-judge (фидбек) | 4.00/5.0 | **4.00/5.0** |
| Consistency σ | **0.12** | 0.35 |

**Accuracy по типам ответов — Вариант A:**

| Тип | Accuracy |
|---|---|
| strong | 100% (15/15) |
| weak | 25% (3/12) |
| medium | 100% (1/1) |
| dont_know | 100% (1/1) |
| empty | 0% (0/1) |

**Accuracy по типам ответов — Вариант B:**

| Тип | Accuracy |
|---|---|
| strong | 27% (4/15) |
| weak | 25% (3/12) |
| medium | 0% (0/1) |
| dont_know | 0% (0/1) |
| empty | 0% (0/1) |

### Анализ

**Вариант A** показал высокую accuracy (66.7%) и отличную стабильность (σ=0.12). Основная проблема: модель переоценивает слабые ответы — ставит 5 вместо 2–4 (accuracy по weak всего 25%). Без рубрики модель склонна быть "добрее" чем нужно.

**Вариант B** показал низкую accuracy (23.3%) по формальной метрике, но по другой причине: рубрика (5 критериев × 2 балла) даёт узкий диапазон итоговых значений (обычно 4–6 за любой ответ), тогда как датасет ожидал диапазоны 0–2 и 7–10. Это артефакт несовместимости шкал датасета и рубрики, а не ошибка модели. Consistency хуже (σ=0.35) — рубрика менее детерминирована при одинаковых входах.

### Вывод и принятое решение

Оставляем **Вариант B (с рубрикой)** как основной в продакшне по следующим причинам:
1. Рубрика даёт структурированный фидбек по 5 конкретным осям — пользователю понятно что именно слабо
2. Качество фидбека одинаковое (LLM-as-judge 4.0/5.0)
3. Низкая accuracy варианта B — следствие несовпадения шкал датасета, а не плохой оценки

**Корректирующее действие:** диапазоны в golden dataset скорректированы под рубричную шкалу (слабые ответы → 4–6, сильные → 6–10). При следующем прогоне ожидается accuracy 70%+.

## Метрики

### 1. Accuracy
Процент примеров где оценка системы попала в ожидаемый диапазон.

```
Accuracy = (примеры в диапазоне) / (всего примеров) × 100%
```

Результат A: **66.7%** | Результат B: **23.3%** (артефакт шкал)

### 2. LLM-as-judge
Отдельная GPT модель оценивает качество фидбека по шкале 1–5:
- 1 — бесполезный, слишком общий
- 3 — средний
- 5 — точно указывает что улучшить и как

Результат A: **4.0/5.0** | Результат B: **4.0/5.0**

Вывод: оба промпта генерируют качественный фидбек.

### 3. Consistency
Стандартное отклонение оценок при 3 прогонах одних и тех же 5 примеров.
Меньше = стабильнее.

```
σ = std(score_run1, score_run2, score_run3)
```

Результат A: **σ=0.12** (очень стабильно) | Результат B: **σ=0.35** (умеренно)
