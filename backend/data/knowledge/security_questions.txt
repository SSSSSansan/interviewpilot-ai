ROLE: Security Engineer

Q: Что такое SQL инъекция и как её предотвратить?
A (strong): SQL инъекция — атака когда пользовательский ввод интерпретируется как SQL код. Пример: поле логина с вводом ' OR '1'='1 -- обходит аутентификацию если запрос строится конкатенацией. Предотвращение: параметризованные запросы / prepared statements (никогда не конкатенировать ввод в SQL), ORM (Django ORM, SQLAlchemy по умолчанию защищены), валидация входных данных, минимальные привилегии для БД пользователя. Обнаружение: ручное тестирование с ', ", --, sqlmap для автоматизации. Code review: искать строковую конкатенацию в SQL запросах.
A (weak): SQL инъекция это когда хакер вводит SQL в форму, нужно валидировать ввод.
Red flags: думает что валидации достаточно (обходится через encoding), не знает параметризованные запросы.

Q: Объясни OWASP Top 10 — назови хотя бы 5 и объясни каждый.
A (strong): 1) Broken Access Control — пользователь получает доступ к чужим данным меняя ID в URL. Защита: проверка прав на каждый запрос серверной стороне. 2) Cryptographic Failures — хранение паролей в plaintext или MD5. Защита: bcrypt/Argon2 для паролей, TLS везде. 3) Injection (SQL, Command, LDAP) — вредоносный ввод выполняется как код. Защита: параметризованные запросы, санитизация. 4) Insecure Design — уязвимости в архитектуре. Защита: threat modeling. 5) Security Misconfiguration — дефолтные пароли, открытые порты, verbose ошибки. Защита: минимальная конфигурация, принцип наименьших привилегий.
A (weak): SQL инъекция и XSS — самые известные уязвимости.
Red flags: знает только 1-2 пункта, не может объяснить как предотвратить, путает OWASP с конкретными атаками.

Q: Что такое XSS и какие виды бывают?
A (strong): XSS (Cross-Site Scripting) — инъекция вредоносного JS в страницу который выполняется в браузере другого пользователя. Виды: Reflected — скрипт в URL, отражается в ответе (фишинговые ссылки). Stored — скрипт сохраняется в БД и показывается всем пользователям (комментарии, профили) — наиболее опасный. DOM-based — клиентский JS записывает данные из URL в DOM. Последствия: кража cookies/токенов, действия от имени пользователя, keylogging. Защита: encode output (htmlspecialchars), Content Security Policy заголовок, HttpOnly флаг на cookies (нельзя украсть через JS).
A (weak): XSS это когда вставляешь JavaScript на страницу.
Red flags: не знает виды XSS, не знает защиту кроме "валидировать ввод".

Q: Объясни как работает аутентификация через JWT.
A (strong): JWT состоит из трёх частей: Header (алгоритм подписи), Payload (claims: user_id, role, exp), Signature (HMAC или RSA подпись). Payload только base64 encoded — не зашифрован, любой может прочитать. Сервер верифицирует подпись при каждом запросе — не хранит сессии (stateless). Типичные ошибки: не проверять подпись, хранить sensitive данные в payload, не проверять expiration, использовать алгоритм "none". Access token — короткий (15 мин), Refresh token — длинный (30 дней), хранится в httpOnly cookie. При компрометации JWT нельзя инвалидировать без blacklist — trade-off stateless архитектуры.
A (weak): JWT это токен для аутентификации, состоит из трёх частей разделённых точкой.
Red flags: думает что payload зашифрован, не знает про верификацию подписи, не понимает проблему инвалидации.

Q: Что такое CSRF и как защититься?
A (strong): CSRF (Cross-Site Request Forgery) — атака где вредоносный сайт заставляет браузер аутентифицированного пользователя отправить нежелательный запрос к целевому сайту (браузер автоматически отправляет cookies). Пример: пользователь залогинен в банке, заходит на evil.com, там форма которая POST на bank.com/transfer. Защита: CSRF токен (случайное значение в скрытом поле формы, сервер проверяет), SameSite=Strict/Lax для cookies (браузер не отправляет cookies в cross-site запросах), проверка Origin/Referer заголовков. Современные SPA с JWT в Authorization header менее уязвимы — браузер не добавляет его автоматически.
A (weak): CSRF это подделка запросов с другого сайта, защита — CSRF токен.
Red flags: не может объяснить механизм атаки, не знает SameSite cookies.

Q: Как безопасно хранить пароли?
A (strong): Никогда: plaintext, MD5, SHA1, SHA256 без соли — все взламываются через rainbow tables или брутфорс. Правильно: bcrypt, Argon2, scrypt — специально медленные хэши, включают соль. bcrypt: cost factor определяет скорость (рекомендуется 10-12, чтобы хэш занимал ~100мс). Аргон2 — победитель Password Hashing Competition, рекомендован для новых проектов. Соль — уникальное случайное значение для каждого пароля, предотвращает rainbow table атаки. Сравнение всегда через timing-safe функцию (hmac_compare_digest) — предотвращает timing атаки.
A (weak): Хранить в зашифрованном виде или хэшировать через SHA256.
Red flags: предлагает MD5/SHA1/SHA256, не знает что такое соль, не знает специализированные алгоритмы.

Q: Объясни что такое penetration testing и его фазы.
A (strong): Pentest — авторизованная симуляция атаки для выявления уязвимостей до того как это сделает злоумышленник. Фазы: 1) Reconnaissance — сбор информации (OSINT: LinkedIn, Shodan, DNS записи). 2) Scanning — port scanning (nmap), service enumeration, vulnerability scanning. 3) Exploitation — эксплуатация найденных уязвимостей (Metasploit, Burp Suite). 4) Post-exploitation — привилегии, lateral movement, что мог бы сделать атакующий. 5) Reporting — документирование с severity, воспроизводимыми шагами, remediation. Всегда только в рамках согласованного scope и с письменным разрешением.
A (weak): Pentest это когда пытаешься взломать систему чтобы найти уязвимости.
Red flags: не знает фазы, не упоминает про авторизацию и scope, путает pentest с vulnerability scanning.

Q: Что такое принцип наименьших привилегий и где он применяется?
A (strong): Principle of Least Privilege (PoLP) — каждый компонент системы должен иметь только минимальные права необходимые для выполнения своей функции. Применение: БД пользователь приложения — только SELECT/INSERT/UPDATE на нужных таблицах, не DROP. IAM роли в AWS — только нужные actions на нужные ресурсы. Сервисные аккаунты — минимальные права. Пользователи — не все сотрудники должны иметь доступ к prod БД. Контейнеры — запускать не от root. Зачем: если компонент скомпрометирован, атакующий ограничен в том что может сделать дальше (ограничение blast radius).
A (weak): Давать пользователям только те права которые им нужны.
Red flags: знает определение но не может привести конкретные примеры в разных контекстах (БД, cloud, OS).
