ROLE: QA Engineer

Q: Объясни пирамиду тестирования и почему она важна.
A (strong): Пирамида тестирования — модель распределения тестов. Основание: unit тесты (70%) — быстрые, дешёвые, тестируют отдельные функции в изоляции. Середина: integration тесты (20%) — тестируют взаимодействие компонентов (API, БД). Верхушка: E2E тесты (10%) — тестируют полные пользовательские сценарии через UI. Чем выше в пирамиде — тем медленнее, дороже и хрупче тест. Инвертированная пирамида (много E2E) — антипаттерн, делает CI медленным и ненадёжным.
A (weak): Нужно писать разные виды тестов — unit, integration и E2E.
Red flags: не знает почему E2E тесты должны быть в меньшинстве, не может объяснить стоимость разных уровней.

Q: Что такое flaky test и как с ним бороться?
A (strong): Flaky test — тест который иногда проходит, иногда падает без изменений в коде. Причины: race conditions (async операции без proper await), зависимость от внешних сервисов, shared state между тестами, hardcoded задержки (sleep). Как бороться: использовать явные ожидания вместо sleep, изолировать тесты (каждый тест независим), мокать внешние зависимости, использовать детерминированные тестовые данные, запускать тест 100 раз чтобы воспроизвести проблему.
A (weak): Flaky test это тест который иногда падает, нужно добавить sleep.
Red flags: предлагает sleep как решение, не понимает изоляции тестов.

Q: Как ты тестируешь REST API?
A (strong): Проверяю: 1) Happy path — корректный запрос возвращает правильный ответ и статус код. 2) Негативные сценарии — невалидные данные возвращают 400 с понятным сообщением, не 500. 3) Аутентификацию — запросы без токена возвращают 401, с недостаточными правами 403. 4) Граничные значения — пустые поля, максимальная длина, спецсимволы. 5) Схему ответа — все обязательные поля присутствуют, типы данных корректны. 6) Производительность — время ответа под нагрузкой. Инструменты: Postman, pytest + requests, Newman для автоматизации.
A (weak): Отправляю запрос в Postman и смотрю что вернулось.
Red flags: тестирует только happy path, не думает о негативных сценариях и безопасности.

Q: Объясни разницу между severity и priority бага.
A (strong): Severity — техническая серьёзность бага (насколько он ломает функциональность): critical (краш, потеря данных), major (ключевая функция не работает), minor (некритичная функция), trivial (косметика). Priority — насколько срочно нужно починить с точки зрения бизнеса. Эти метрики независимы. Пример высокий severity / низкий priority: краш в редко используемой фиче для 0.1% пользователей. Пример низкий severity / высокий priority: опечатка в названии компании на главной странице.
A (weak): Severity это насколько серьёзный баг, priority это когда чинить.
Red flags: не может привести пример когда они расходятся, путает эти понятия.

Q: Как ты подходишь к тест-дизайну когда нет требований?
A (strong): Использую exploratory testing. Сначала изучаю продукт как пользователь — понимаю бизнес-логику. Применяю техники: equivalence partitioning (делю входные данные на классы), boundary value analysis (тестирую границы). Проверяю: happy path, основные error cases, security basics (XSS, SQL injection в полях ввода), UX (понятны ли сообщения об ошибках). Документирую что тестировал и что нашёл. Отсутствие требований — тоже находка: поднимаю как риск.
A (weak): Тестирую всё подряд и записываю баги.
Red flags: нет системного подхода, не знает техник тест-дизайна, не документирует процесс.

Q: Что такое regression testing и когда его запускать?
A (strong): Regression testing — проверка что новые изменения не сломали существующую функциональность. Когда запускать: после каждого изменения кода (автоматически в CI), перед релизом, после hotfix в продакшне. Стратегии: полная регрессия (долго, перед крупным релизом), частичная (тесты связанных компонентов, при каждом PR), smoke тесты (критичные пути, быстро, при каждом деплое). Без автоматизации регрессия становится узким местом всего процесса.
A (weak): Регрессия это когда проверяешь что ничего не сломалось.
Red flags: не понимает как приоритизировать регрессионный набор, не думает об автоматизации.

Q: Как ты измеряешь качество тестового покрытия?
A (strong): Code coverage — процент строк/веток кода покрытых тестами. 100% coverage не означает отсутствие багов — можно покрыть строки но не проверить правильность логики. Более важны: покрытие бизнес-требований (все acceptance criteria протестированы), покрытие рисков (высокорисковые области протестированы лучше), мутационное тестирование (проверяет что тесты действительно ловят изменения в коде). Coverage — метрика-направление, не цель сама по себе.
A (weak): Смотрю на процент code coverage, чем выше тем лучше.
Red flags: считает 100% coverage достаточным, не знает других метрик качества тестирования.

Q: Расскажи про performance testing — какие виды бывают и что измеряешь?
A (strong): Load testing — тестирование под ожидаемой нагрузкой. Stress testing — нагрузка выше нормы, ищем точку отказа. Spike testing — резкий скачок нагрузки. Soak/Endurance testing — длительная нагрузка для выявления утечек памяти. Ключевые метрики: response time (среднее, p95, p99), throughput (запросов в секунду), error rate, утилизация ресурсов (CPU, память). Инструменты: k6, JMeter, Locust. Результаты сравниваю с baseline и SLA требованиями.
A (weak): Performance testing это проверка что сайт не тормозит под нагрузкой.
Red flags: знает только load testing, не понимает разницу между средним и перцентилями latency.
