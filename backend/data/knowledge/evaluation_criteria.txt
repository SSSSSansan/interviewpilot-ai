EVALUATION CRITERIA AND RED FLAGS BY ROLE

---

ОБЩИЕ КРИТЕРИИ СИЛЬНОГО ОТВЕТА (все роли):
- Кандидат отвечает структурированно: сначала суть, потом детали
- Использует конкретные примеры, а не абстрактные описания
- Знает trade-offs: понимает когда подход хорош, а когда нет
- Может объяснить просто: избегает жаргона без необходимости
- Честен о пробелах: "не знаю, но вот как бы я разбирался"

ОБЩИЕ RED FLAGS (все роли):
- Отвечает только buzzwords без понимания ("использую AI/ML/микросервисы")
- Не может привести ни одного конкретного примера из своего опыта
- Агрессивно защищает один подход без рассмотрения альтернатив
- Замалчивает провалы, представляет только успехи
- Использует "мы делали" без объяснения своей личной роли

---

BACKEND ENGINEER — специфичные критерии:

Сильный кандидат:
- Думает о масштабировании и производительности
- Знает когда добавить кэш, индекс, очередь
- Понимает безопасность: SQL injection, auth, HTTPS
- Пишет тестируемый код, знает разницу unit/integration/e2e тестов
- Знает про мониторинг и логирование

Red flags:
- Не думает о production: только "работает на моей машине"
- Не знает что такое транзакция
- Путается в HTTP методах (использует GET для изменений)
- Не понимает зачем нужны миграции БД

---

FRONTEND ENGINEER — специфичные критерии:

Сильный кандидат:
- Думает о пользовательском опыте, не только о коде
- Понимает accessibility (ARIA, keyboard navigation)
- Знает про performance: lazy loading, bundle size, Core Web Vitals
- Умеет дебажить в DevTools

Red flags:
- Не думает о mobile/responsive
- Не знает что такое SEO implications JS рендеринга
- Игнорирует кросс-браузерную совместимость
- Не знает разницу between == и ===

---

ML ENGINEER — специфичные критерии:

Сильный кандидат:
- Начинает с данных, а не с модели
- Думает о деплое и мониторинге модели в продакшне
- Знает про data leakage и как его избежать
- Понимает вычислительную стоимость решений

Red flags:
- Считает что нейросеть — решение любой задачи
- Не знает про train/val/test split
- Не думает о воспроизводимости (reproducibility)
- Не умеет работать с imbalanced data

---

DATA ANALYST — специфичные критерии:

Сильный кандидат:
- Всегда начинает с бизнес-вопроса, не с данных
- Скептичен к данным: проверяет качество и источник
- Умеет объяснить результаты нетехнической аудитории
- Знает про correlation vs causation

Red flags:
- Использует средние без проверки на выбросы
- Принимает данные как данность без проверки качества
- Строит красивые дашборды без бизнес-вопроса
- Делает выводы о причинности из корреляции

---

PRODUCT MANAGER — специфичные критерии:

Сильный кандидат:
- Всегда начинает с проблемы пользователя, не с решения
- Думает о метриках до начала работы над фичей
- Умеет говорить "нет" с обоснованием
- Понимает технические ограничения

Red flags:
- Говорит "я бы добавил фичу" без объяснения проблемы которую решает
- Не знает ни одной продуктовой метрики
- Воспринимает roadmap как обещание, а не план
- Не может объяснить как работает продукт технически

---

ШКАЛА ОЦЕНКИ ОТВЕТОВ:

9-10 (Excellent): Полный ответ с примерами, знает trade-offs, может углубиться в любой аспект
7-8 (Good): Правильный ответ, понимает суть, возможно упустил нюансы
5-6 (Acceptable): Базовое понимание, но поверхностный ответ без деталей
3-4 (Weak): Частично верно, много пробелов, нет практического понимания
1-2 (Poor): Существенные ошибки или почти нет понимания темы
0 (No answer): Не знает совсем / отказался отвечать

is_weak = true если score < 6 → задаём follow-up вопрос
