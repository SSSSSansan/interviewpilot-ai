ROLE: Data Analyst

Q: Объясни разницу между JOIN типами в SQL.
A (strong): INNER JOIN — только совпадающие строки из обеих таблиц. LEFT JOIN — все строки левой + совпадения из правой (NULL если нет). RIGHT JOIN — все строки правой + совпадения из левой. FULL OUTER JOIN — все строки из обеих, NULL где нет совпадений. CROSS JOIN — декартово произведение (каждая с каждой). На практике LEFT JOIN чаще всего: хочу все заказы даже без клиента.
A (weak): JOIN соединяет таблицы, INNER только совпадения.
Red flags: не знает FULL OUTER JOIN, не может объяснить на примере когда какой использовать.

Q: Что такое оконные функции в SQL?
A (strong): Оконные функции вычисляются по "окну" строк без схлопывания результата (в отличие от GROUP BY). OVER(PARTITION BY col ORDER BY col) — определяет окно. Примеры: ROW_NUMBER() — порядковый номер, RANK() — ранг с пропусками, LAG/LEAD — значение предыдущей/следующей строки, SUM() OVER — скользящая сумма. Используются для: рейтингов, скользящих метрик, сравнения с предыдущим периодом.
A (weak): Оконные функции это как GROUP BY но по-другому.
Red flags: не знает синтаксис OVER PARTITION BY, не может назвать примеры использования.

Q: Как ты проверяешь качество данных перед анализом?
A (strong): Чеклист: 1) Null values — % пропущенных по колонкам, стратегия заполнения. 2) Дубликаты — COUNT vs COUNT DISTINCT. 3) Выбросы — boxplot, IQR метод, бизнес-смысл. 4) Типы данных — дата записана как string? 5) Диапазоны значений — отрицательный возраст? 6) Референсная целостность — все FK существуют? 7) Временные аномалии — пробелы в временном ряду. Фиксирую находки в data quality report.
A (weak): Смотрю на данные и проверяю что нет ошибок.
Red flags: не знает про выбросы, не делает системную проверку.

Q: Объясни разницу между средним, медианой и модой. Когда что использовать?
A (strong): Среднее — сумма/количество, чувствительно к выбросам. Медиана — центральное значение, устойчива к выбросам. Мода — наиболее частое значение. Когда что: зарплаты/доходы → медиана (Билл Гейтс искажает среднее), нормально распределённые данные → среднее, категориальные данные → мода, оценка производительности системы → p95/p99 (не среднее!).
A (weak): Среднее это сумма делённая на количество, медиана это середина.
Red flags: не знает когда медиана лучше среднего, не упоминает выбросы.

Q: Как построить A/B тест правильно?
A (strong): 1) Формулировка гипотезы H0/H1. 2) Выбор метрики (primary + guardrail). 3) Расчёт sample size (power analysis: effect size, α=0.05, power=0.8). 4) Рандомизация без загрязнения (сегрегация пользователей). 5) Минимальная длительность: минимум 1-2 недели (seasonality). 6) Анализ: t-test для нормальных, Mann-Whitney для ненормальных. 7) Проверить guardrail метрики не упали. 8) Принятие решения с учётом практической значимости (не только p-value).
A (weak): Разделяем пользователей на две группы и сравниваем конверсию.
Red flags: не считает sample size заранее, останавливает тест при первом p<0.05 (peeking problem).

Q: Что такое когортный анализ и когда он нужен?
A (strong): Когортный анализ — группировка пользователей по дате первого действия (регистрация, первая покупка) и отслеживание их поведения во времени. Нужен для: измерения retention (% вернувшихся через N дней), сравнения поколений пользователей (улучшился ли онбординг?), расчёта LTV. Пример: когорта январь 2024 vs февраль 2024 — как изменился 30-day retention после редизайна онбординга?
A (weak): Когортный анализ это когда группируешь пользователей.
Red flags: не знает для чего нужен retention анализ, не может привести конкретный пример.
