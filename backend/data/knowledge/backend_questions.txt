ROLE: Backend Engineer

Q: Объясни разницу между SQL и NoSQL базами данных.
A (strong): SQL — реляционные БД с фиксированной схемой и ACID транзакциями (PostgreSQL, MySQL). Гарантируют целостность данных, хорошо подходят для финансов, e-commerce. NoSQL — гибкая схема, горизонтальное масштабирование (MongoDB, Redis, Cassandra). Лучше для больших объёмов неструктурированных данных, логов, кэша. Выбор зависит от задачи: если нужны транзакции и JOIN — SQL, если нужна скорость и масштаб — NoSQL.
A (weak): SQL использует таблицы, NoSQL нет.
Red flags: не может назвать конкретные БД, не знает CAP теорему, не может объяснить когда что выбрать.

Q: Что такое индекс в БД и когда его не стоит добавлять?
A (strong): Индекс — вспомогательная структура данных (B-tree, Hash) которая ускоряет SELECT за счёт дополнительного хранилища и замедления INSERT/UPDATE/DELETE. Не стоит добавлять на: колонки с низкой кардинальностью (boolean, status с 3 значениями), таблицы с частыми записями где reads редки, маленькие таблицы где full scan быстрее.
A (weak): Индекс ускоряет поиск, добавляй везде где медленно.
Red flags: не знает что индекс замедляет запись, не может объяснить B-tree.

Q: Что такое REST и какие у него ограничения?
A (strong): REST — архитектурный стиль для API на HTTP. Принципы: stateless (каждый запрос самодостаточен), uniform interface (HTTP методы GET/POST/PUT/DELETE), ресурсы по URL. Ограничения: overfetching (получаешь лишние поля), underfetching (нужно несколько запросов), нет встроенной типизации. Альтернативы: GraphQL решает overfetching, gRPC даёт типизацию и скорость.
A (weak): REST это когда API через HTTP.
Red flags: путает REST и HTTP, не знает stateless принцип.

Q: Объясни разницу между процессом и потоком.
A (strong): Процесс — независимая единица выполнения со своим адресным пространством памяти. Потоки — лёгкие единицы внутри процесса, разделяют память. Потоки быстрее создавать и переключать, но race conditions опаснее. В Python GIL ограничивает параллельность потоков для CPU-задач — там лучше multiprocessing. Для I/O задач потоков достаточно.
A (weak): Процесс это программа, поток это часть программы.
Red flags: не знает про GIL в Python, не понимает разницу CPU-bound vs IO-bound.

Q: Что такое Docker и зачем он нужен?
A (strong): Docker — платформа контейнеризации. Контейнер упаковывает приложение со всеми зависимостями в изолированную среду. Решает проблему "у меня работает, у тебя нет". Отличие от VM: контейнеры делят ядро ОС → легче и быстрее. Основные концепции: Image (шаблон), Container (запущенный image), Dockerfile (инструкция сборки), docker-compose (многоконтейнерные приложения).
A (weak): Docker это чтобы запускать приложения в контейнерах.
Red flags: не знает разницу image vs container, не может объяснить зачем нужен.

Q: Как работает HTTP и что такое HTTPS?
A (strong): HTTP — протокол передачи данных клиент-сервер на базе TCP. Запрос содержит метод, URL, заголовки, тело. Ответ — статус код, заголовки, тело. HTTPS = HTTP + TLS шифрование: защищает от перехвата, аутентифицирует сервер через сертификат. TLS handshake: клиент и сервер договариваются о ключах шифрования перед передачей данных.
A (weak): HTTPS это защищённый HTTP с замочком.
Red flags: не знает статус коды (200, 404, 500), не понимает разницу GET и POST.

Q: Что такое микросервисы и их минусы?
A (strong): Микросервисы — архитектура где приложение разбито на независимые сервисы, каждый отвечает за свою бизнес-область, деплоится и масштабируется отдельно. Плюсы: независимый деплой, технологическая свобода, изоляция отказов. Минусы: сложность distributed systems (network latency, partial failures), сложность отладки и трейсинга, overhead на межсервисное взаимодействие. Не нужны для маленьких команд и простых продуктов.
A (weak): Микросервисы это когда много маленьких сервисов вместо одного большого.
Red flags: не видит минусов, считает что микросервисы всегда лучше монолита.
